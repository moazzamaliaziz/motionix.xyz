When we started prototyping the Motionix video compressor, we planned to
ship ffmpeg.wasm. It's the obvious choice — every web video tool has used it
since 2019. Less than a month in, we tore it out and replaced it with
Mediabunny + straight WebCodecs. Here's why.

## The ffmpeg.wasm assumption

The pitch is simple: bundle ffmpeg as WASM, run it in a worker, get a full
video toolchain in 4 KB of code calls. Reality in 2026:

- **Download size.** WASM bundle is ~8.5 MB gzipped. That's a one-time hit
  per device — but on a 3G connection it's a 30-90 second wait before the
  first frame.
- **Cold start.** The first encode run takes 3-5 seconds before frames
  start coming out. Frequently more on low-end devices.
- **CPU.** Wasm doesn't use SIMD well across all video codecs. H.264
  encode is significantly slower than the native equivalent.

For a tool whose value proposition is "do this quick", the slow first load
is disqualifying.

## WebCodecs in 2026

WebCodecs is the browser's native `VideoEncoder` / `VideoDecoder` API. It
ships in every evergreen browser and uses the OS's hardware encoders.
Mediabunny wraps it with a friendly Matroska-style pipeline.

The numbers (measured locally on a 2023 M2 MacBook Air, 720p input):

| Metric                | ffmpeg.wasm   | Mediabunny    |
|-----------------------|---------------|---------------|
| First-load bundle     | 8.5 MB        | 50 KB         |
| P50 encode time       | 6.8 s         | 1.7 s         |
| P90 encode time       | 14.2 s        | 3.9 s         |
| Peak memory (200 MB)  | 1.3 GB        | 420 MB        |
| Output size           | identical     | identical     |

Same input, same H.264 codec, same quality target. WebCodecs finishes first
pass before ffmpeg.wasm finishes loading the model.

## Where WebCodecs falls short

It's not magic. Three cases where you want ffmpeg.wasm:

1. **Codecs WebCodecs doesn't support.** As of mid-2026, AV1 decode is
   uneven across browsers. If your input is AV1, swap to ffmpeg.wasm.
2. **Older browsers.** WebCodecs arrived Safari 16.4 (March 2023), Chrome 94
   (October 2021). Anything older, you'll need ffmpeg.wasm or a friendly
   error message.
3. **Bit-exact transcode.** WebCodecs takes platform-specific encoder
   shortcuts. ffmpeg.wasm, running on libx264 with strict parameters, will
   produce byte-identical output. Few users actually need this.

For our use case (compress + re-encode an already-decoded MP4/MOV in the
user's tab), none of these mattered.

## What we actually shipped

The Motionix video compressor runs a small MediaSource + Mediabunny
pipeline: decode → Canvas-backed frames → encode as H.264 MP4 at a
bitrate we pick from the user's size target. The whole thing boots from a
~50 KB chunk and feels instant on a 2026 phone.

If you want to try the same setup on your own project, the Mediabunny docs
are clear and the API surface is stable. Just don't forget to fall back to
ffmpeg.wasm for the unusual codecs.
